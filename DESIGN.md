for the Rust reference southbound **EtherNet/IP** adapter. Implementation agents build to THIS
document; deviations must be surfaced to the user before landing, not documented after the fact.
This is an internal doc — status, roadmap, and decision-register content belong here and must NOT
leak into the user-facing `docs/` set (which describes current behavior only, present tense).

CIP protocol crate** specified in [`PROTOCOL-DESIGN.md`](PROTOCOL-DESIGN.md) (not `rseip` —

I/O **push** (D-EIP-2). The repo becomes a Cargo **workspace**: `crates/enip` (the protocol
library, isolated from everything EdgeCommons) + `crates/ethernet-ip-adapter` (this adapter).
Protocol wire details live ONLY in `PROTOCOL-DESIGN.md`; this document owns adapter behavior.

## 1. Overview & scope

`ethernet-ip-adapter` is the **Rust reference southbound protocol adapter** for
**EtherNet/IP** — the CIP (Common Industrial Protocol) encapsulation over TCP/IP used by
Allen-Bradley ControlLogix / CompactLogix PLCs and many generic CIP devices. It is the third
reference adapter (after OPC UA / Java and Modbus / Python) and the first in Rust, proving the
southbound contract is language-complete and exercising **two** corners of the design space at
once: the **tag-addressed, poll-based** model (explicit messaging, like Modbus) and the
**device-push** model (class-1 implicit I/O — cyclic produced/consumed assemblies at an RPI,
the industrial-Ethernet analog of OPC UA's subscribe side). It rides the **owned protocol stack**
(`crates/enip`, [`PROTOCOL-DESIGN.md`](PROTOCOL-DESIGN.md)) rather than an external protocol
library.

### 1.1 EtherNet/IP + CIP background (what an implementer must know)

- **EtherNet/IP** encapsulates CIP over TCP port **44818** (explicit messaging) and UDP 2222
  (implicit class-1 I/O). A TCP session is opened with an EtherNet/IP *RegisterSession*, then CIP
  requests ride *SendRRData* (unconnected) or *SendUnitData* (connected, after a CIP
  *ForwardOpen*).
- **Explicit messaging** is request/response: the client sends a CIP service (Read Tag `0x4C`,
  Write Tag `0x4D`, Get Instance Attribute List `0x55` for tag discovery) addressed by an
  **EPath**. This is the adapter's **poll** mode — scheduled polling, like the Modbus reference.
- **Implicit I/O** (class-1 cyclic UDP produced/consumed assemblies) is the device-push model:
  the adapter (as scanner/originator) ForwardOpens an I/O connection against the device's
  assembly instances; the device then produces its input assembly cyclically at the negotiated
  RPI over UDP :2222, and the adapter produces an output assembly (or heartbeat) back. This is
  the adapter's **push** mode. Assemblies are raw bytes; config maps byte offsets to typed,
  named signals (§4.6). Mode is selected **per instance** (`mode: poll | push`, D-EIP-2).
- **The AB tag model**: Logix controllers expose *tags* — named, typed variables
  (`LINE_SPEED : REAL`, `ZONE_TEMPS : REAL[8]`, program-scoped
  `Program:MainProgram.FillTimer.ACC`). A tag is addressed by its symbolic path (the EPath). CIP
  elementary types carry explicit type codes (DINT `0xC4`, REAL `0xCA`, …). Reads/writes to a
  ControlLogix CPU in a chassis are routed through a **connection path** (backplane port 1 +
  CPU slot, e.g. slot 0); devices without a backplane (CompactLogix directly, our simulator) use
  an empty/default path.
- In EdgeCommons vocabulary a data point is a **signal** (never "tag" — that word is reserved for
  envelope business metadata). A CIP tag maps to a signal: `signal.id` is its stable tag path,
  `signal.address` the protocol-native object (§5).

### 1.2 Where it sits in the site topology

One component instance (`component.instances[]` entry) = **one device** (one PLC / one CIP
endpoint), each with its own task, session, and connection lifecycle — the template's
one-task-per-instance supervisor, kept as-is. The adapter publishes `SouthboundSignalUpdate` on
the device-local bus (`data` class); `uns-bridge` uplinks to the site broker; `edge-console`
consumes the site UNS and drives the adapter through its `cmd` inbox (panels in §7.6). In the
Dallas bottling harness it joins `bottling-company-test/` as a southbound source alongside the
Modbus and OPC UA adapters, fed by the cpppo (poll) and OpENer (push) simulators (§11).

### 1.3 In scope / out of scope

**In scope (v1):** the **owned protocol stack** (`crates/enip`, its own design doc); **poll
mode** — scheduled polling of config-declared signals (scalars + arrays) over CIP explicit
messaging (unconnected and connected class-3); **push mode** — class-1 implicit I/O
(produced/consumed assemblies at an RPI, P2P and multicast consume, run/idle, sequence/stale/
timeout validation) with config-declared assembly layouts; confirmed allow-listed writes in both
modes; the full `sb/*` command family incl. the **`sb/pause` / `sb/resume` contract extension**
(per-instance, both modes); CIP tag discovery (`sb/browse`) where the device supports it; full
metric-family parity with Modbus/OPC UA plus a push-mode I/O family; per-instance connectivity +
paused reflection; deadband/on-change publishing with batching; decoder **fuzzing**; Diátaxis
docs; 90% coverage gate (workspace-wide); all three platform packs.

**Out of scope (v1, stated in §14):** UDT/structure values; Logix STRING (a UDT); CIP Multiple
Service Packet request batching; CIP Security/TLS; acting as a class-1 *target* in production
(the test harness ships a minimal target, PROTOCOL-DESIGN §12.5); store-and-forward (Tier-2 core
concern); editing core `SOUTHBOUND.md` (pause/resume promotion is *proposed*, not performed, by
this work).

## 2. Decisions register

| # | Decision | Rationale / alternatives |
|---|---|---|
| **D-EIP-1** | **Protocol stack = the OWNED pure-Rust `crates/enip` crate ([`PROTOCOL-DESIGN.md`](PROTOCOL-DESIGN.md)). No external protocol library.** *(Supersedes v1.0's "use rseip 0.3".)* | Both vetted OSS options failed the bar: `rseip` is frozen, panics on a truncated reply, has UTF-8 UB in tag listing, and correlates connected replies with a `debug_assert!` only (silent wrong-tag values in release builds); `EIPScanner` (C++) overruns on runt I/O frames and swallows sequence/size validation — and neither is maintained. On a wire where every inbound byte is device/attacker-controlled, we own the stack: memory-safe bounds-checked decoding (`#![forbid(unsafe_code)]`), real reply correlation with per-request deadlines, validated class-1 consumption, fuzzed decoders. Both references were studied for wire-format correctness (they agree with each other and with ODVA on every layout); the code is original. Pure Rust ⇒ builds natively on Windows/MSVC, Linux, and the Greengrass WSL path with zero C deps. The 2026-07-18 cpppo de-risk probe stands as *wire* validation (read `LINE_SPEED`, wrote+read back `FILL_SETPOINT`); its rseip API pattern is retired with this decision. |
| **D-EIP-2** | **Update model = BOTH: `mode: "poll"` (scheduled explicit-messaging polling, the default) and `mode: "push"` (class-1 implicit I/O), selected per instance.** *(Supersedes v1.0's "poll-only, implicit I/O unsupported" — that limit was rseip's, and rseip is gone.)* | Poll mirrors the Modbus poll-group model (groups + deadband + on-change publish). Push consumes the device's cyclic input assembly at the RPI and maps configured byte-offset fields to signals (§4.6); publish gating (deadband/onChange/batch/sample floor) still applies so a 10 ms RPI does not flood the UNS. One instance = one device = one mode; a plant that wants both from one PLC configures two instances. |
| **D-EIP-3** | **`sb/pause` / `sb/resume` are a deliberate southbound-contract EXTENSION.** | Neither reference adapter has them; `SOUTHBOUND.md` §2.2 does not list them. They are designed fully here (§7.4, §9) and are a **candidate for promotion into core `SOUTHBOUND.md`** as a poll-adapter convention — but this work does NOT edit core. Surfaced up front per the fidelity contract. |
| **D-EIP-4** | **Naming: repo/crate/bin `ethernet-ip-adapter`; component `com.mbreissi.edgecommons.EthernetIpAdapter`; UNS component token `ethernet-ip-adapter` (via the config `component.token` override); `device.adapter` = `"ethernet-ip"`.** | Matches the sibling kebab convention (`modbus-adapter`, `opcua-adapter`). The CLI mangled the crate name to `ethernetipadapter` with no override (CLI-DOGFOODING #1/#2); the current tree still carries that name in `Cargo.toml`/`Dockerfile`/`recipe.yaml`/`compose.yaml`/`supervisor/` — renaming them all is implementation slice S1. `component.token` is a shipped core feature (`config/identity.rs`: `configured_component_token` overrides the PascalCase short name), so the UNS token is `ethernet-ip-adapter` without renaming the Greengrass component. |
| **D-EIP-5** | **Writes are allow-listed and secure-by-default: empty `writes.allow[]` ⇒ ALL writes refused, matched on the stable `signal.id`.** | Keeps the template's gate verbatim (checked in the command handler before the write reaches the device task) and the D-U16 convention. Deliberately *not* Modbus's legacy boolean `write.enabled` — the allow-list is the target contract (`SOUTHBOUND.md` §2.2) and strictly stronger. |
| **D-EIP-6** | **Two live simulators, one per mode: cpppo (AB tag server, :44818) for POLL; an OpENer target container for PUSH (owned in-crate test target as the CI fallback).** The in-process sim backend stays and is extended to model both. | Unit tests reach 90% with no PLC and no network (sim backend + the protocol crate's mock peer); E2E and sim-gated integration tests talk to the real wire (§11). cpppo cannot serve class-1 implicit I/O, so push needs a real target: OpENer is the independent-implementation conformance peer; the `enip` crate's `testserver` target covers CI and offline dev (§11.5). Coverage: the protocol crate is fully unit-testable, so only the sim-gated live suites stay outside the denominator (§12.2). |
| **D-EIP-7** | **Signals are declared explicitly in config poll groups; `sb/browse` is on-demand discovery, not subscription matching.** | Modbus-style declaration (no regex include/exclude like OPC UA — that model belongs to a subscription protocol). `sb/browse` uses the Logix tag-list service (Get Instance Attribute List, `EipClient::list_tags`) so a console can discover tags; generic CIP devices may not implement it ⇒ `BROWSE_UNSUPPORTED` (§7.3, §14.4). |
| **D-EIP-8** | **Poll mode: `connection.connected: false` is the default (unconnected explicit messaging); `true` opens a class-3 CIP connected session (ForwardOpen).** | Unconnected is the simplest path and what cpppo serves. Connected class-3 (PROTOCOL-DESIGN §7.6) allocates a connection for higher-rate polling against real PLCs — supported, config-selected, validated in the lab (not against cpppo). Push mode always uses its own class-1 ForwardOpen and ignores this key. |
| **D-EIP-9** | **`signal.id` = the configured `tagPath`, verbatim (case-sensitive).** | One instance = one device, so the tag path is unique within the instance and stable across restarts — no synthetic prefix needed (`u<unit>/…` exists in Modbus because unit id is real addressing there; slot is connection routing, not identity, so it stays out of the id and inside `signal.address`). The UNS topic channel comes from the config `name` (lower-kebab), NOT from the id — §5.3/§6.1. |
| **D-EIP-10** | **Quality: per-signal read failure ⇒ published `BAD` sample with the CIP status in `qualityRaw`; connection-level failure ⇒ leave the poll loop and reconnect; `UNCERTAIN` for scale/coercion anomalies.** | Keeps the template's "a failed read is published, not swallowed" rule and the §3 normalization table (§5.4). |
| **D-EIP-11** | **Timestamps: `serverTs` = adapter read time (facade default "now"); `sourceTs` is NEVER emitted.** | CIP explicit read replies carry no device timestamp. The facade never synthesizes `sourceTs`, honoring `SOUTHBOUND.md` §2 ("at least one SHOULD be present" — `serverTs` is). |
| **D-EIP-12** | **Metric families: mandatory `southbound_health` (+ a `paused` gauge) PLUS `EtherNetIpConnection` / `EtherNetIpInventory` / `EtherNetIpPoll` / `EtherNetIpPublish` / `EtherNetIpCommand` / `EtherNetIpIo` (push), every counter as a `(…Total, …Interval)` measure pair.** | Family structure and dimensions mirror Modbus (`Modbus*` families, dims `instance`/`pollGroup`/`result`/`verb`/`publishMode`); the explicit Total/Interval measure-pair spelling mirrors OPC UA (`ReadRequestTotal`/`ReadRequestInterval`). `EtherNetIpIo` is the push-mode addition (frames, sequence gaps, stale drops, watchdog timeouts — §8.8), surfacing the validation counters the protocol stack keeps (PROTOCOL-DESIGN §8.6). This is the parity superset — full tables in §8. Dimensions are low-cardinality ONLY; tag names/addresses/endpoints are never dimensions. |
| **D-EIP-13** | **Command routing: all verbs registered ONCE on the component-scope inbox; the target device rides `body.instance`, optional when exactly one device is configured.** | Exactly the Modbus pattern (`main.py` registers each verb once and resolves by the body's `instance` selector) and the template's `sb/write`. One inbox serves every device this adapter owns (D-U28). |
| **D-EIP-14** | **Pause semantics (poll mode): pause suspends the *scheduled poll/publish* only. The session stays open, on-demand `sb/read`/`sb/write` still work, and a slow keepalive probe keeps `connected` driven by real CIP I/O.** (Push-mode counterpart: D-EIP-20; the mode-uniform contract — pause suspends the instance's telemetry production — is normative in §7.4.) | The alternative (tear the session down) conflates "operator muted the data flow" with "link is down" and would make `connected` a lie in both directions. Full semantics in §7.4/§9.3. Pause state is **in-memory** — a component restart resumes production (documented; a persistent disable would be config, not a command). |
| **D-EIP-15** | **One CIP request per signal per poll cycle (no Multiple Service Packet batching) in v1.** | MSP batching is a performance optimization with real encoding complexity; with the owned stack it is now *feasible* (a natural `enip` crate v2 feature) but still deliberately deferred. Stated as a limitation (§14.3) with the `EtherNetIpInventory.requestsPerCycle` measure making the cost visible. Push mode does not have this cost (one frame carries the whole assembly). |
| **D-EIP-16** | **Supported value types: CIP elementary scalars + 1-D arrays thereof. `string`/UDT/structs are rejected at config validation.** | The owned codec decodes every elementary type (§5.1 incl. usint/uint/udint/ulint/lreal); Logix STRING is a UDT and struct *values* are out of scope for v1 (the stack detects and reports them — PROTOCOL-DESIGN §6.3). Rejecting at startup with a clear error beats a runtime surprise (§5.2, §14.2). Applies to both poll `type` and push field `type`. |
| **D-EIP-17** | **The repo is a Cargo workspace: `crates/enip` (protocol library) + `crates/ethernet-ip-adapter` (this binary). The protocol crate knows NOTHING about EdgeCommons** — no UNS, envelopes, metrics, commands, config schema, or edgecommons dependency; the adapter consumes it only through the `device.rs` seam. | The isolation is what makes the stack independently testable, fuzzable, and reusable (PROTOCOL-DESIGN §1). Mirrors the `core/cli` workspace convention. The old "thin rseip call seam excluded from coverage" carve-out dies with this: the protocol crate is inside the 90% gate. |
| **D-EIP-18** | **Push signals are config-declared assembly-layout fields** (`byte offset + type [+ bit] [+ arrayCount]` within the input/output assembly), with stable `signal.id` = `a<assemblyInstance>/<offset>/<type>[.<bit>]`. | Raw I/O is just bytes — the layout IS the signal model (the Modbus register-map analog for push). The id form mirrors Modbus's `u<unit>/<table>/<addr>/<type>`: derived from real addressing, stable across restarts, independent of the display `name`. Bounds are validated at startup by the crate's `AssemblyLayout` (construction-time, so runtime extraction cannot go out of bounds). |
| **D-EIP-19** | **Decoder fuzzing is a release gate**: the protocol crate's cargo-fuzz targets (PROTOCOL-DESIGN §12.3) run in CI on every PR (short budget, checked-in corpus + regressions); a reproducible panic/OOM found by fuzzing is a ship blocker. | The memory-safety claim (D-EIP-1's reason to exist) must be executable, not aspirational. Fuzzing runs on Linux (CI/WSL — libFuzzer). |
| **D-EIP-20** | **Pause in push mode keeps the I/O connection OPEN**: O→T production continues (heartbeat/outputs, run/idle unchanged), consumed frames still update the last-value snapshot and drive connectivity — but nothing is published to `data`. | Per-instance pause suspends *telemetry production* whichever mode the instance runs (user clarification). Closing the connection would conflate "operator muted the data flow" with "link is down" (same rationale as D-EIP-14), and stopping O→T production would make the *target* time the connection out (PROTOCOL-DESIGN D-ENIP-9). Push needs no keepalive probe: T→O reception itself is the liveness signal. |

## 3. Module architecture

### 3.1 Source layout (workspace — D-EIP-17)

Keep the template's seam and supervisor; add the protocol crate, config, metrics, poll/push,
and command modules. Nothing above the seam learns the protocol; nothing below it learns the
UNS. The protocol crate's internals are specified in `PROTOCOL-DESIGN.md` §3 — not here.

```text
Cargo.toml                  workspace root (resolver 3; members = crates/*)
crates/
  enip/                     THE OWNED PROTOCOL CRATE (see PROTOCOL-DESIGN.md — its layout,
                            fuzz/ targets, tests/vectors/, testserver are specified there)
  ethernet-ip-adapter/      the adapter binary crate:
    src/
      main.rs        entry point (kept; binary renamed ethernet-ip-adapter, S1)
      app.rs         supervisor (kept + extended): per-device task spawn (poll OR push engine
                     per instance mode), connectivity provider, command wiring, pause plumbing,
                     health emit cadence
      device.rs      the protocol seam (kept + extended): DeviceBackend / DeviceSession /
                     PushSession traits, Reading, IoUpdate, Quality, DeviceError

                     the cpppo tag layout AND a scripted input-assembly producer + failure
                     injection hooks)
      eip/
        mod.rs       EipBackend: DeviceBackend impl (connect → EipSession / open_push →
                     EipPushSession) over the enip crate's EipClient / IoManager
        session.rs   EipSession: DeviceSession impl — poll reads, subset reads, writes, browse,

        push.rs      EipPushSession: PushSession impl — forward_open from config, IoEvent →

                     batching windows, stale tracking, pause gate
      push.rs        the per-device PUSH engine: IoUpdate consumption, per-field change/deadband
                     + sample-floor gating, batching, stale/timeout tracking, pause gate,

      publish.rs     SignalUpdate assembly (data() facade calls), publish-latency measurement
```

### 3.2 The loop, and where everything hooks in

```text
             ┌────────────────────────────────────────────────────────────────────┐
             │                        run_device (one task per device)            │
             │                                                                    │
   config ──►│  connect (EipBackend::connect, timeout connectMs)                  │
             │    │ ok: LinkState::Online, evt device-connected,                  │
             │    │     clear-alarm device-unreachable, health emit_now           │
             │    ▼                                                              │
             │  poll loop (poll.rs)  ◄────────────────────────────┐              │
             │    ├─ tokio::select! over:                          │              │
             │    │    • per-group tickers (pollIntervalMs)        │              │
             │    │    • write channel (WriteRequest from cmd)     │              │
             │    │    • control channel (Pause/Resume/Repoll/     │              │
             │    │      Reconnect/ReadNow from cmd)               │              │
             │    │    • keepalive probe ticker (only while paused)│              │
             │    ├─ paused? ─ yes → skip group ticks (writes,     │              │
             │    │            on-demand reads, probe still run)   │              │
             │    ├─ read each signal (1 CIP request each)         │              │
             │    │    per-signal failure → BAD Reading            │              │
             │    │    connection failure → break ──────────────┐  │              │
             │    ├─ deadband/change gate + batch window         │  │              │
             │    ├─ publish via data() facade (publish.rs)      │  │              │
             │    └─ counters → metrics.rs                       │  │              │
             │                                                   │  │              │
             │  on link break: LinkState::Backoff, reconnects++, │  │              │
             │  evt raise-alarm device-unreachable, health now ◄─┘  │              │
             │    │                                                 │              │
             │    ▼                                                 │              │
             │  backoff (exponential, full jitter, cap; permanent → ceiling) ─────┘
             └────────────────────────────────────────────────────────────────────┘

  cmd inbox (component scope, commands.rs) ── body.instance ──► the device's write/control
  channels; sb/status & the connectivity provider read the shared Health/Paused state.
```

The template's nested connect/poll loops, the single-task write serialization, and the
backoff are **kept**. New: the control channel (pause/resume/repoll/reconnect/read-now travel the
same one-task serialization path as writes), per-group tickers instead of one, and the paused
gate.

**The push variant (`mode: "push"`)** runs the same `run_device` supervisor with the poll loop
replaced by the push engine (`push.rs`): connect = TCP session + class-1 ForwardOpen
(`PushSession::open`); the `tokio::select!` arms become **the IoUpdate event stream** (instead of
group tickers), the write channel (output-assembly field writes), and the control channel. Per
consumed frame: decode fields via the validated layout → per-field change/deadband +
`sampleMs` floor gate → batch → publish (same `publish.rs`). `IoEvent::Lost` (watchdog timeout,
peer close) breaks into the same backoff/reconnect ladder; `paused` gates *publishing only* —
consumption, snapshot updates, and O→T production continue (D-EIP-20). O→T production and
connection liveness live inside the protocol crate (PROTOCOL-DESIGN §8.6–§8.7); the engine only
consumes typed events.

### 3.3 The seam, extended (device.rs)

The two template traits are kept and extended. The boundary rule stands: **a backend knows
protocols; it never imports `edgecommons::uns`, topics, envelopes, or metrics.**

```rust
#[async_trait]
pub trait DeviceSession: Send + Sync {
    /// Read the given configured signals once (a poll group's worth, or an sb/read subset).
    /// Per-signal failure => that signal comes back as a BAD Reading; Err(_) only when the
    /// CONNECTION is broken.
    async fn read_signals(&mut self, signals: &[SignalSpec]) -> Result<Vec<Reading>>;

    /// Write one value (already coerced/validated by the codec) to a signal. Confirmed:
    /// Ok(()) means the device acknowledged the CIP write.
    async fn write_signal(&mut self, signal: &SignalSpec, value: &serde_json::Value) -> Result<()>;

    /// Enumerate device tags (CIP Get Instance Attribute List), one page.
    /// Default impl: Err(DeviceError::Unsupported) — the sim implements it; generic CIP
    /// devices may not.
    async fn browse(&mut self, cursor: Option<String>, max: usize) -> Result<BrowsePage>;

    /// A minimal liveness probe (used by the paused keepalive): cheapest real CIP round-trip
    /// the backend can do (v1: read of the instance's first configured signal).
    async fn probe(&mut self) -> Result<()>;

    /// Close the session. Safe to call twice.
    async fn close(&mut self);
}

pub struct BrowsePage {
    pub tags: Vec<BrowsedTag>,          // { name, type_name, array_dim: Option<u32>, instance_id: u32 }
    pub next_cursor: Option<String>,    // None => last page
}
```

`DeviceError` gains one variant beyond the template's `Transient`/`Permanent`:
`Unsupported(&'static str)` — "this device/backend does not implement the operation" (browse on
a non-Logix device). It is neither retried nor a link failure.

Changes from the template signature (surfaced deliberately): `read_signals` now takes the signal
list (the template's zero-arg version hid the config inside the session; poll groups and
`sb/read` subsets need parameterized reads), and `write_signal` takes the resolved `SignalSpec`
(the codec needs the type to build the wire value). The sim backend is updated to match.

**The push side of the seam.** `DeviceBackend` gains `open_push(&IoSpec) → Result<Box<dyn
PushSession>>` (poll instances never call it; push instances never call `connect`):

```rust
#[async_trait]
pub trait PushSession: Send + Sync {
    /// The consumed-I/O stream: decoded field updates + connection lifecycle.
    /// The engine owns the receiver; the session owns translation from the transport.
    fn updates(&mut self) -> &mut mpsc::Receiver<IoUpdate>;

    /// Write one output-assembly field (already coerced by the codec). Ok(()) means the
    /// field is set in the producer buffer and will ride the next O→T frame (§7.3 honesty note).
    async fn write_field(&mut self, signal: &SignalSpec, value: &serde_json::Value) -> Result<()>;

    /// Snapshot of the last consumed input assembly (for sb/read while paused/without waiting).
    fn last_input(&self) -> Option<InputSnapshot>;   // decoded fields + received_at + run_mode

    async fn close(&mut self);                       // ForwardClose + socket teardown; idempotent
}

pub enum IoUpdate {
    Up   { o2t_api_ms: u32, t2o_api_ms: u32 },
    Data { fields: Vec<(SignalKey, serde_json::Value)>, run_mode: bool, received_at: Instant },
    Lost { error: DeviceError },                     // watchdog timeout / peer close / IO error
}
```

The boundary rule is unchanged: the backend maps the protocol crate's types (`IoEvent`,
`CipValue`, `EnipError`) into seam types; the engines above the seam never see the `enip` crate.
`browse`/`probe` remain on `DeviceSession` (explicit messaging) — a push instance that is also a
Logix device can still serve `sb/browse` through a short-lived explicit session (§7.5).

### 3.4 The enip backend (eip/) — consuming the owned stack

`eip/` is the only adapter module that imports the `enip` crate, against the **public API pinned
in PROTOCOL-DESIGN §11.2** (implementation agents: that section is the contract — do not
improvise around it):

```rust
use enip::{EipClient, ClientOptions, RoutePath, TagAddress, CipType, CipValue,
           IoManager, IoConnectionSpec, IoEvent, AssemblyLayout, EnipError};

// poll (unconnected default; connected class-3 when connection.connected = true):
let client = EipClient::connect(&endpoint, ClientOptions {
    route: slot.map(RoutePath::backplane_slot),      // absent slot ⇒ None (cpppo/CompactLogix)
    connect_timeout, request_timeout,                // from §4.1 timeouts
    connected_messaging: cfg.connection.connected,
    ..Default::default()
}).await?;
let r = client.read_tag(&TagAddress::parse(&spec.tag_path)?, spec.array_count.unwrap_or(1)).await?;
// r.wire_type is compared against the configured EipType — mismatch ⇒ BAD ("DECODE type mismatch")
client.write_tag(&addr, spec.eip_type.into(), &coerced).await?;   // Ok = CIP-acked (confirmed)

// push:
let io  = IoManager::bind(("0.0.0.0", 2222)).await?;              // one per process, shared
let conn = io.forward_open(&client, spec.io_connection_spec()).await?;  // from §4.6 config
conn.set_output(&initial_output)?;  conn.set_run(true);
// conn.events() → IoEvent::{Up, Data, Lost} → translated to IoUpdate via the AssemblyLayout
```

- **Timeouts and correlation live in the crate** (per-request deadlines, stale-reply quarantine,
  class-1 watchdog — PROTOCOL-DESIGN §10); the adapter maps `EnipError` → `DeviceError` using
  `EnipError::is_transient()` as the default, overridden per §10.1's table.
- **Typed decode**: `read_tag` returns the wire-declared `CipValue`; `eip/types.rs` converts to
  JSON per the configured `EipType` and flags mismatches as BAD samples (never a decode panic —
  the crate guarantees totality, D-ENIP-4).
- One request per signal per cycle in poll mode (D-EIP-15).
- The `AssemblyLayout` for a push instance is built **once at startup** from §4.6 config;
  construction validates every field against the assembly size, so a bad offset is a config
  error at start, not a runtime surprise (D-EIP-18).

| Key | Type / default | Meaning |
|---|---|---|
| `name` | string, required, lower-kebab (UNS-token pattern) | Human label AND the `data` topic channel (§6.1). Unique per device. |
| `tagPath` | string, required | The CIP tag path, verbatim/case-sensitive (`"LINE_SPEED"`, `"Program:Main.FillPV"`). It IS the stable `signal.id`. Unique per device. |
| `type` | enum, required | `bool, sint, usint, int, uint, dint, udint, lint, ulint, real, lreal` (§5.2). Required — CIP decode is typed. |
| `arrayCount` | int ≥ 1, optional | Present ⇒ the signal is a 1-D array read of that many elements (`read_tag(&tag, count)`); value is a JSON array. |

refuses to start). Additional startup validations: duplicate `name`/`tagPath` within a device;
`writes.allow[]` entries that match no configured `tagPath` (poll) / no output field id (push)
log a warning (they may be intentional for `sb/write`-by-explicit-ref, §7.2 — warned, not
rejected); `scale`/`deadband` on `bool` rejected.

## 5. Signal & data-type model

### 5.1 CIP → JSON value mapping

| Config `type` | CIP type (code) | Rust decode | JSON value | Write accepts |
|---|---|---|---|---|
| `bool` | BOOL (0xC1) | `bool` | `true`/`false` | JSON boolean |
| `sint` | SINT (0xC2) | `i8` | number | integral number in range |
| `usint` | USINT (0xC6) | `u8` | number | integral number in range |
| `int` | INT (0xC3) | `i16` | number | integral number in range |
| `uint` | UINT (0xC7) | `u16` | number | integral number in range |
| `dint` | DINT (0xC4) | `i32` | number | integral number in range |
| `udint` | UDINT (0xC8) | `u32` | number | integral number in range |
| `lint` | LINT (0xC5) | `i64` | number | integral number in range |
| `ulint` | ULINT (0xC9) | `u64` | number | integral number in range |
| `real` | REAL (0xCA) | `f32` | number | any JSON number |
| `lreal` | LREAL (0xCB) | `f64` | number | any JSON number |

- **Arrays** (`arrayCount: N`): poll reads request N elements (`read_tag(&tag, N)`); push fields
  extract N contiguous elements from the assembly. The JSON value is a JSON array of N elements,
  each per the row above (`SOUTHBOUND.md` §2 value-typing rule).
  A write to an array signal accepts a JSON array of exactly N elements, each coerced to the
  element type; length mismatch is a per-entry write failure (`"expected N elements"`).
- **scale/offset** (numeric only): read: `published = raw * scale + offset` (f64 arithmetic;
  the published JSON number is f64). Write: `device = (value - offset) / scale`, then
  range-checked and coerced to the CIP type; out-of-range ⇒ per-entry failure, never a clamp.
  Element-wise for arrays.
- **Rejected at config validation** (D-EIP-16): `string`, structures/UDTs, multi-dimensional
  arrays. The simulator's `RECIPE=SSTRING` tag exists precisely to prove the adapter's
  validation/browse story for unsupported types (§11.1).

### 5.2 Stable identity split (`SOUTHBOUND.md` §2)

| Contract field | Poll mode | Push mode |
|---|---|---|
| `device.adapter` | `"ethernet-ip"` (`"sim"` for the simulator backend) | same |
| `device.instance` | the instance id (`"filler-plc"`) | same (`"palletizer-io"`) |
| `device.endpoint` | `connection.endpoint` (`"127.0.0.1:44818"`) | same |
| `signal.id` | the `tagPath`, verbatim — e.g. `"LINE_SPEED"`, `"Program:Main.FillPV"` (D-EIP-9) | `a<assemblyInstance>/<offset>/<type>[.<bit>]` — e.g. `"a100/0/bool.0"`, `"a100/4/udint"` (D-EIP-18) |
| `signal.name` | the config `name` (`"line-speed"`) | same (`"motor-run"`) |
| `signal.address` | `{ "tagPath": "ZONE_TEMPS", "type": "real", "arrayCount": 8, "slot": 0 }` — optional keys only when configured | `{ "assembly": 100, "offset": 0, "type": "bool", "bit": 0, "arrayCount": 7, "slot": 0 }` — optional keys only when configured |
| `samples[]` | one per publish; batched under `batchMs` (§6.2) | same |

`signal.address` is the protocol-native round-trip object: everything needed to re-address the
value (tag path + decode type + array length + routing slot for poll; assembly instance + byte
offset + type + bit for push). Both id forms are derived from real, restart-stable addressing —
there is no volatile-index problem here (unlike OPC UA `ns=`), so id and address agree by
construction. The push id mirrors Modbus's `u<unit>/<table>/<addr>/<type>` shape.

### 5.3 Topic channel vs id

The `data` topic channel is the config **`name`** (lower-kebab, already a valid UNS token), set
explicitly via `SignalUpdateBuilder::signal_path(name)`. The raw `tagPath` (with `:`, `.`,
uppercase) rides only in the body (`signal.id`), untouched. This keeps topics clean without
lossy sanitization of the id (the facade would otherwise sanitize the id into the channel).

### 5.4 Quality mapping (`SOUTHBOUND.md` §3)

| Normalized | Condition (poll) | Condition (push) | `qualityRaw` |
|---|---|---|---|
| `GOOD` | CIP read succeeded, value decoded | field decoded from an accepted frame with run-mode `Run` (or no O→T header semantics apply) | `"0x00"` (CIP general status Success) |
| `BAD` | CIP error response for THIS tag (e.g. path segment error 0x04/0x05 = tag not found; service not supported 0x08; insufficient data 0x13), or decode/type mismatch | — (a mis-sized/malformed frame is dropped+counted by the stack, never decoded into samples) | the typed `CipStatus` rendering, e.g. `"0x04 (path segment error)"`; decode failures `"DECODE <detail>"` |
| `BAD` | per-request timeout that did NOT break the session (isolated) | — | `"TIMEOUT"` |
| `UNCERTAIN` | value decoded but the scale/offset produced a non-finite number (NaN/inf) | same | `"NON_FINITE_AFTER_SCALE"` |
| `UNCERTAIN` | — | frame accepted but the peer signals **Idle** in the run/idle header (values present, process not running) | `"IDLE"` |

A BAD sample's `value` is JSON `null`. Connection-level failures (socket errors, RegisterSession
loss, repeated timeouts — poll; class-1 watchdog timeout / `IoEvent::Lost` — push) are NOT
samples — they break the engine loop into reconnect (§10), and the signals' staleness accounting
takes over.

`serverTs`: poll = adapter read time (facade default, injected clock); push = the frame's
`received_at`, so batched samples keep their true arrival times. `sourceTs` never emitted in
either mode — neither explicit replies nor class-1 frames carry a device timestamp (D-EIP-11).

### 6.2 The `SouthboundSignalUpdate` publish path (exact facade sequence)

Per changed signal per publish flush (verified against `facades/data.rs` /
`facades/signal_update.rs` at the pinned rev):

```rust
let mut sample = Sample::with_quality(value, quality);       // Quality::{Good,Bad,Uncertain}
sample = sample.quality_raw(raw);                            // ALWAYS set (never rely on the
                                                             // "unspecified" default marker)
// batching: additional Samples get an explicit .server_ts(read_iso) so a coalesced batch
// preserves per-read times; the newest sample may leave server_ts to the facade default.

let update = data.signal(&spec.stable_id)       // poll: tagPath (D-EIP-9); push: a<inst>/<off>/<type> (D-EIP-18)
    .name(&spec.name)
    .address(spec.address_json())               // §5.2 object
    .device_parts("ethernet-ip", &cfg.id, &cfg.connection.endpoint)
    .signal_path(&spec.name)                    // topic channel = name (§5.3)
    .samples(batch)                             // 1..N samples (batchMs window)
    .build();
data.publish(update).await?;                    // header SouthboundSignalUpdate/1.0, identity
                                                // stamped, topic minted — all facade-owned
```

- **Batching**: with `batchMs > 0`, samples for one signal are buffered and flushed when the
  window closes (per-signal windows, opened by the first buffered sample). `publishMode:
  onChange` gates *entry into* the buffer (deadband); `always` buffers every poll. A BAD sample
  always passes the gate (a failure is information).
- **Publish latency**: wall time of the `data.publish().await` call, recorded into
  `southbound_health.publishLatencyMs` and `EtherNetIpPublish.publishLatencyMs`.
- Channel routing (local ▸ config `publish.channel` ▸ per-call `via`) is left at the facade
  default (LOCAL); the config passthrough works without adapter code if a deployment sets
  `publish.channel`.

### 6.3 Stateful `evt` surface

Via the `events()` facade only (verified signatures: `emit`, `raise_alarm`, `clear_alarm` with
`Severity` + type + message + context):

| Event `type` | Kind / severity | When | `context` |
|---|---|---|---|
| `device-connected` | emit, Info | session established | `{instance, adapter, endpoint}` |
| `device-unreachable` | **alarm**, Critical: `raise_alarm` on link loss, `clear_alarm` on re-establish | link transitions | `{instance}` (+ `error` on raise) |
| `adapter-paused` | emit, Warning | `sb/pause` took effect | `{instance, by: <requester identity path if present>}` |
| `adapter-resumed` | emit, Info | `sb/resume` took effect | `{instance}` |
| `write-audit` | emit, Info (success) / Warning (failure or refusal) | every `sb/write` entry, INCLUDING allow-list refusals | `{instance, signalId, ok, value, error?}` |

Raise/clear ride the same severity so they share one channel (facade rule, template behavior
kept).

## 7. Command surface

All verbs are registered on the component's `CommandInbox` (`gg.commands()`), advertised by the
built-in `describe`, and are **request/reply** (a fire-and-forget request still executes; the
reply is dropped by the inbox). The library-owned built-ins (`ping`, `describe`, `status`,
`reload-config`, `get-configuration`) come for free and are not re-implemented.

**Instance routing (D-EIP-13):** every verb accepts `"instance": "<id>"` in the body. It is
optional iff exactly one device is configured (then it defaults to that device); with ≥ 2 devices
a missing/unknown instance is `NO_SUCH_INSTANCE`. Replies always include the resolved
`"id": "<instance>"`.

**Error codes** (reply `{"ok":false,"error":{"code","message"}}` via `CommandError`):
`BAD_ARGS`, `NO_SUCH_INSTANCE`, `WRITE_NOT_ALLOWED`, `WRITE_FAILED`, `DEVICE_UNAVAILABLE`
(device task gone / channel closed), `READ_FAILED` (connection-level failure during an on-demand
read), `RECONNECT_FAILED`, `BROWSE_UNSUPPORTED`, `BROWSE_FAILED`. Per-entry problems inside
batch read/write are reported inline in the result, not as command errors.

### 7.1 `sb/status` — instance status

Request: `{ "instance"? }` →

```json
{ "id": "filler-plc", "mode": "poll", "connected": true, "state": "ONLINE", "paused": false,
  "endpoint": "127.0.0.1:44818", "adapter": "ethernet-ip",
  "metrics": { "read": { "interval": 412, "total": 15230 },
               "write": { "interval": 1, "total": 12 },
               "readErrors": { "interval": 0, "total": 3 } } }
```

A push instance answers the same shape plus an `io` object:
`"io": { "o2tApiMs": 100, "t2oApiMs": 100, "run": true, "peerRun": true,
"framesConsumed": {...}, "staleDropped": {...}, "sequenceGaps": {...} }` (negotiated APIs from
the ForwardOpen reply, run/idle both directions, and the §8.8 counters).

`connected`/`state`/`paused` come from the SAME shared Health/Paused state the connectivity
provider reads — ask or watch, same answer. (The universal `status` built-in already returns
`instances[]`; `sb/status` is the southbound-detail superset for one instance.)

### 7.2 `sb/read` — on-demand batch read

Request: `{ "instance"?, "signals": [ <signal-ref>, … ] }` where **`<signal-ref>`** is
`{"name": "<configured name>"}` (friendly) **or** `{"tagPath": "...", "type": "<EipType>",
"arrayCount"?}` (explicit, for tags not in config — the Modbus explicit-ref analog). Reply:

```json
{ "id": "filler-plc", "reads": [
    { "signal": { "id": "LINE_SPEED", "address": { "tagPath": "LINE_SPEED", "type": "real" } },
      "value": 54.9, "quality": "GOOD", "qualityRaw": "0x00",
      "serverTs": "2026-07-18T12:00:00Z" } ] }
```

Reads reuse the §5 value/quality encoding exactly (`SOUTHBOUND.md` §2.2). Unresolvable refs come
back as entries with `"quality":"BAD"`, `"qualityRaw":"UNRESOLVED_REF"`. Reads are serviced by
the device task (a `ReadNow` control message carrying a oneshot reply), so they serialize with
the engine loop — and they work while paused. On-demand reads are **not** published to `data`.

**Push instances:** `sb/read` answers from the **last consumed input snapshot**
(`PushSession::last_input`) — no device round-trip exists for "read one field now" in implicit
I/O. Each entry's `serverTs` is the snapshot's receive time; if the connection is down or no
frame has arrived yet, entries answer `"quality":"BAD"`, `"qualityRaw":"NO_FRAME"`. The explicit
push ref form is `{"assembly": 100, "offset": 4, "type": "udint", "bit"?, "arrayCount"?}` —
resolved against the configured layout only (arbitrary offsets outside the declared
`sizeBytes` are `UNRESOLVED_REF`; the layout is the contract).

### 7.3 `sb/write` — confirmed, allow-listed batch write

Request: `{ "instance"?, "writes": [ { <signal-ref>, "value": <any> }, … ] }` (a single object
without the `writes` wrapper is also accepted, per §2.2). Reply:

```json
{ "id": "filler-plc", "written": 1, "results": [
    { "signal": "FILL_SETPOINT", "value": 55.5, "ok": true },
    { "signal": "MOTOR_SPEED", "ok": false, "error": "not in writes.allow" } ] }
```

Normative gate order, per entry: resolve ref → **allow-list check against the stable
`signal.id`** (tagPath or push field id; refusal = inline `ok:false` + a `write-audit` Warning
event; the whole command errors `WRITE_NOT_ALLOWED` only when ALL entries are refused) → codec
coercion (§5.1; failure inline) → the write travels the device task's write channel (template
`WriteRequest` kept, extended to carry the `SignalSpec`) → **confirmed**: `ok:true` only after
the CIP reply. Every entry is audited on `evt` (§6.3). An explicit-ref write is still matched
against `writes.allow[]` by its id — the allow-list gates identity, not config membership.

**Push instances** write **output-assembly fields** (input fields are never writable — refusal
`"error": "input field"`). Confirmation honesty (stated verbatim in user docs): implicit I/O has
no per-write device acknowledgment; `ok:true` means *the field is set in the O→T producer buffer
and rides every subsequent cyclic frame* (`PushSession::write_field`). The reply carries
`"applied": "next-frame"` on push entries so consoles can render the distinction from poll's
CIP-acked writes.

### 7.4 `sb/pause` / `sb/resume` — the contract extension (D-EIP-3, D-EIP-14)

Request: `{ "instance"? }`. Replies: `{ "id", "paused": true|false, "changed": true|false }` —
**idempotent** (pausing a paused instance is `changed:false`, never an error).

Normative pause semantics — **per-instance, mode-uniform: pause suspends the instance's
*telemetry production* (its `data` publishes), whether it polls or consumes push I/O** (user
clarification; D-EIP-14 + D-EIP-20):

1. **No `data` publishes occur while paused.** Poll: scheduled poll ticks are gated OFF (all
   groups). Push: consumed frames are NOT published — but consumption itself continues (the
   snapshot, sequence validation, and connectivity stay live).
2. The **session/connection stays open**. Poll: the explicit session. Push: the class-1
   connection — O→T production continues at the API (heartbeat/outputs, run/idle unchanged;
   stopping would make the *target* time us out, D-EIP-20). `sb/read`, `sb/write`, `reconnect`,
   `sb/browse` still work.
3. `connected` stays truthful via real I/O. Poll: a **keepalive probe** (`DeviceSession::probe`,
   cadence `healthThresholds.keepaliveProbeIntervalMs`) since no polls flow; a probe failure
   drives the normal reconnect path (still paused after reconnecting). Push: **no probe needed**
   — T→O reception and the watchdog already are the liveness signal.
4. Reflection — all three surfaces move together: (a) `state` keepalive `instances[]`: `state`
   token becomes **`"PAUSED"`**, `attributes.paused: true` (`connected` stays truthful);
   (b) metric: `southbound_health.paused` gauge = 1 (§8.1); (c) events: `adapter-paused` /
   `adapter-resumed` (§6.3).
5. Staleness accounting is **suspended** while paused (§9.3) — a paused signal is paused, not
   stale.
6. Pause state is in-memory; restart ⇒ production resumes (documented in `docs/`).
7. `repoll` against a paused instance is refused with `BAD_ARGS`
   (`"instance is paused - resume first"`) — an explicit one-shot read is what `sb/read` is for.
8. On resume, push change-detection state is **re-based to the current snapshot** so the paused
   span's accumulated drift is not published as one giant "change" burst; poll behaves likewise
   via its per-signal last-published values.

> **Contract note (surfaced deliberately):** `sb/pause`/`sb/resume` extend beyond
> `SOUTHBOUND.md` §2.2 and beyond both reference adapters. They are designed here as a
> general poll-adapter convention and are a candidate for promotion into core `SOUTHBOUND.md`
> (a core-repo change, out of scope for this work — do NOT edit core here).

## 8. Metrics

All defined via `MetricBuilder::create(name).with_config(&config).add_measure(...)
.add_dimension(...)` and emitted through the injected `Arc<dyn MetricService>` — destination is
config-driven (`metricEmission.target`), no adapter code per target. Units are `Count`,
`Milliseconds`, `None`; resolution 1 (high) only for live gauges, else 60.

**Conventions (normative):**
- Every **counter** is a measure PAIR: `<name>Total` (monotonic since component start) and
  `<name>Interval` (since the previous emit of that family; reset on emit). The OPC UA pair
  spelling over the Modbus family/dimension structure — D-EIP-12.
- **Gauges** (`connectionState`, `sessionConnected`, `paused`, `batchSize`, latencies,
  inventory sizes) are single measures.
- Dimensions are **low-cardinality only**: `instance`, `pollGroup`, `result`
  (`success`|`error`), `verb` (closed set §8.6), `publishMode` (`onChange`|`always`),
  `connectionMode` (`connected`|`unconnected`). **Never** tag names, signal ids, addresses,
  endpoints, or error text. (`coreName`/`category`/`component` are injected by the builder.)
- One metric definition per (family × dimension combination), defined at startup like Modbus
  (`_define_*`), values accumulated in atomics/mutexed counters, emitted on the cadence (§8.7).

## 9. State, connectivity & pause reflection

### 9.1 The provider (kept from the template, extended)

ONE `InstanceConnectivityProvider` registered in `App::run`
(`gg.set_instance_connectivity_provider`), one entry per configured device, feeding BOTH the
`state` keepalive's `instances[]` and the built-in `status` verb — push and pull cannot
disagree. Sample shape (verified `InstanceConnectivity` API: `new(instance, connected, detail)
.with_state(token).with_attributes(map)`):

```json
{ "instance": "filler-plc", "connected": true, "state": "PAUSED",
  "detail": "127.0.0.1:44818",
  "attributes": { "adapter": "ethernet-ip", "mode": "poll", "paused": true, "slot": 0,
                  "connectionMode": "unconnected" } }
```

(A push instance reports `"mode": "push"` and `"connectionMode": "class1-io"`; `connected`
means the class-1 connection is open and inside its watchdog window.)

### 9.2 `LinkState` gains `Paused`

```
Connecting ──connect ok──► Online ◄──resume── Paused
     ▲                      │  ▲                │
     └──────backoff◄──break─┘  └────pause───────┘        (break while paused → Backoff,
                                                          reconnect, then back to Paused)
```

Tokens: `CONNECTING` / `ONLINE` / `BACKOFF` / `PAUSED`. The paused flag is stored separately
from the link state (an `AtomicBool` beside `Health`) so a link break while paused reports
`BACKOFF` + `attributes.paused: true`, and re-establishment returns to `PAUSED` (not `ONLINE`).
The reported `state` token is `PAUSED` whenever the instance is paused AND the session is up;
`connected` always tells the truth independently. The template invariant is preserved and
extended: `connected`, `southbound_health.connectionState`, and the state token all derive from
the same `Health` object — and `paused` (attribute, gauge, token) derives from the same
`AtomicBool` — so no two surfaces can disagree.

### 9.3 Connectivity is driven by real CIP I/O

`Health::set_link` is called only from actual I/O outcomes. Poll: connect result, poll-cycle
result, write result, on-demand read result, and — while paused — the keepalive probe (§7.4.3).
Push: ForwardOpen result, `IoUpdate::Up` (first accepted frame), and `IoUpdate::Lost` (watchdog
timeout / peer close) — the cyclic reception is itself the health signal, paused or not. A
cached socket handle is never consulted. Staleness: per-signal `last_good_read: Instant` (poll:
last GOOD read; push: last accepted frame carrying the field), counted against `staleSignalSecs`
at health-emit time; the clock **suspends during pause** (on resume, `last_good_read` is
re-based so the paused span doesn't count).

## 10. Error handling & reconnect

### 10.1 Classification (`EnipError` → `DeviceError`)

The protocol crate's `EnipError::is_transient()` (PROTOCOL-DESIGN §10.1) is the default; the
backend overrides per this table (which is the adapter's normative behavior):

| Failure | Class | Behavior |
|---|---|---|
| TCP connect refused/unreachable, host-lookup failure, connect timeout (`Io`/`Timeout` on connect) | `Transient` | backoff ladder (a device that is off may come back) |
| Malformed endpoint/config (unparsable host, bad slot, layout/`AssemblyLayout` rejection) | `Permanent` | config-time where possible; else backoff at the ceiling (template rule: never hammer what cannot fix itself) |
| Socket error / session loss / EOF mid-cycle (`ConnectionLost`) | `Transient` | leave engine loop → reconnect |
| ≥ `requestTimeoutMs` timeout on a request (`Timeout`) | first: BAD sample (`"TIMEOUT"`); **3 consecutive** timeouts: the stack itself declares the session lost → `Transient` | isolated slowness ≠ dead link; sustained silence = dead link (the stale-reply quarantine guarantees a late answer can never leak into a newer request — PROTOCOL-DESIGN §10.4) |
| CIP error status for one tag (`Cip`: 0x04/0x05 not-found, 0x08, 0x13, …) | not an error — **BAD Reading** | published; session lives (one dead tag must not blind the other ninety-nine) |
| ForwardOpen rejected (`ForwardOpenRejected`) — class-3 (poll `connected: true`) or class-1 (push) | `Transient` | reconnect ladder; the typed extended status is logged (duplicate connection 0x0100, out of connections 0x0113, bad path 0x0315 …); after 3 consecutive rejections log a mode-specific hint (poll: try `connected: false`; push: check assembly instances/sizes against the device) |
| Class-1 watchdog timeout / peer stops producing (`IoEvent::Lost`) | `Transient` | leave push loop → reconnect (ForwardClose best-effort first); `ioTimeouts` counter + `device-unreachable` alarm |
| Malformed/hostile peer traffic (`Malformed`, `ProtocolViolation`) | `Permanent`-leaning: surface loudly, backoff at ceiling | a peer that breaks the protocol will keep breaking it; per-frame UDP garbage is dropped+counted by the stack without touching the link |
| Browse service unsupported (`Cip` 0x08 on the tag-list service) | `Unsupported` | `BROWSE_UNSUPPORTED` reply; never affects the link |

### 10.2 Reconnect

The template's backoff is kept verbatim: exponential from `reconnectBackoffMinMs`, full jitter,
capped at `reconnectBackoffMaxMs`; permanent failures jump to the ceiling. On every
loss: `LinkState::Backoff`, `reconnects`/`connectionDrops` counters, `device-unreachable` alarm
raise; on re-establishment: alarm clear, `device-connected` event, counters — and **pause
survives reconnect** (§9.2). Writes/reads arriving while disconnected fail fast with
`DEVICE_UNAVAILABLE` (the control/write channels are drained with errors during backoff rather
than queueing into a dead link).

### 10.3 Shutdown

Template behavior kept: `gg.shutdown_signal().await` → sessions closed (`close()` is
idempotent — poll: UnRegisterSession; push: ForwardClose + socket teardown, both deadline-bounded
in the protocol crate so shutdown never hangs), final `flush_metrics()`; the library handles
unsubscribes and the `STOPPED` state (the org unsubscribe-before-exit rule is satisfied by RAII).

```bash
  FILL_SETPOINT=REAL ZONE_TEMPS=REAL[8] MOTOR_RUN=DINT RECIPE=SSTRING
```

| Tag | Type | Role in validation |
|---|---|---|
| `LINE_SPEED`, `FILL_TEMP`, `TANK_LEVEL` | REAL | fast-group scalars; deadband + scale/offset paths |
| `PRODUCT_COUNT` | DINT | integer decode path |
| `ZONE_TEMPS` | REAL[8] | **array** read → JSON array of 8; array write shape |
| `FILL_SETPOINT` | REAL | **writable** setpoint (in `writes.allow[]`) — confirmed write + poll-observed readback |
| `MOTOR_RUN` | DINT (0/1) | writable command-style value (cpppo models BOOL awkwardly; DINT stands in — a real Logix BOOL uses `type: bool`) |
| `RECIPE` | SSTRING | deliberately UNSUPPORTED type: proves config validation rejects `string` and `sb/browse` marks it `supported: false` |

cpppo has no backplane ⇒ config omits `slot` (`PortSegment::default()`); the `slot` path is a

### 11.3 Sim-gated integration tests (`tests/live_cpppo.rs`, `tests/live_opener.rs`)

Self-skipping: a `TcpStream::connect` probe (`127.0.0.1:44818` / `:44819`) with a short timeout
at suite start; on failure every test returns early with an eprintln "skipped (no sim)" —
exactly the sibling pattern (Modbus's live-slave gating). Covered live paths — poll
(`live_cpppo.rs`): connect; scalar/array/DINT reads with exact values seeded via cpppo; write +
readback of `FILL_SETPOINT`; per-tag BAD on a nonexistent tag while others stay GOOD; browse
paging incl. `RECIPE` marked unsupported; reconnect after the sim's TCP drop (kill/restart
container is manual — the automated test covers timeout-driven transient classification only).
Push (`live_opener.rs`): ForwardOpen against the §4.6 assemblies; cyclic consume ≥ N frames
with fields decoding to expected values; output write observed to change the produced O→T frame
(verified via OpENer's mirrored output where available, else via a scanner-side capture);
watchdog timeout classification when the container pauses. Both suites excluded from the
coverage denominator (§12.2). *(The protocol crate's own mock-peer tests — PROTOCOL-DESIGN
§12.2 — are NOT gated and run everywhere.)*

### 11.4 E2E validation plan (HOST, scripted; the "does it really work" pass)

Stack: EMQX (`localhost:1883`) + cpppo + OpENer + the adapter (`--platform HOST --transport
MQTT`), configured with BOTH instances: `filler-plc` (poll, §4.5) and `palletizer-io` (push,
§4.6). Ordered checks, each observed on the bus (MQTTX/mosquitto_sub over the §6.1 topics) or
via request/reply:

1. **Poll → data**: `ecv1/+/ethernet-ip-adapter/filler-plc/data/#` carries
   `SouthboundSignalUpdate` for the fast group at ~500 ms; `zone-temps` value is a JSON array
   of 8; identity element stamped; quality GOOD with `qualityRaw: "0x00"`.
2. **Push → data**: `ecv1/+/ethernet-ip-adapter/palletizer-io/data/#` carries updates sourced
   from cyclic frames (respecting `sampleMs`/deadband, NOT one publish per 100 ms frame);
   `signal.id`s are the `a100/…` forms; `serverTs` tracks frame arrival.
3. **State**: keepalive `instances[]` shows both instances `connected: true` (`ONLINE`), the
   push one with `connectionMode: "class1-io"`.
4. **sb/read**: poll — by name and explicit ref, values matching a parallel cpppo client read;
   push — snapshot values with the frame-arrival `serverTs`.
5. **sb/write**: poll — `FILL_SETPOINT` → 55.5 confirmed `ok:true`, next poll publishes 55.5,
   `write-audit` event observed; non-allow-listed refusal + Warning audit + `writeFailures`
   increment. Push — `dout-word` write `ok:true` with `applied: "next-frame"`, and the O→T
   frame observably changes; a write to an input field refused inline.
6. **sb/pause / sb/resume (both modes)**: pause each instance → its `data` traffic stops; state
   token `PAUSED`; `southbound_health.paused`=1; `adapter-paused` event; `sb/read` still
   answers; **push: `framesConsumed`/`framesProduced` keep counting while paused** (the
   connection lives, D-EIP-20); resume → traffic returns, all three surfaces flip back, no
   burst of stale "changes" (§7.4.8).
7. **Metrics**: with `metricEmission.target: messaging`, the `metric` class shows all seven
   families (§8.1–§8.8) with the §8 measure names; Total counters monotonic, Interval counters
   reset; `EtherNetIpIo.interFrameMs` ≈ the negotiated API.
8. **Failure — poll**: stop cpppo → `device-unreachable` alarm raise, `connectionState` 0,
   BACKOFF state; restart → clear + recovery.
9. **Failure — push**: `docker pause` opener-sim → within `timeoutMultiplier × API` the
   watchdog fires: `IoEvent::Lost`, alarm raise, `ioTimeouts` increments, BACKOFF; `docker
   unpause` → ForwardOpen re-establishes, alarm clears, frames resume.
10. **Browse**: `sb/browse` on `filler-plc` lists the 8 tags, `RECIPE` `supported: false`,
    `configured` flags correct.

### 11.5 The class-1 push target (OpENer) — and the fallback

**Recommendation: OpENer** (`EIPStackGroup/OpENer`, the ODVA-member OSS EtherNet/IP
*adapter/target* stack) as the live push target: it is an independent implementation (a genuine
conformance check, not our own code talking to itself), implements class-1 produced/consumed
assemblies with the standard demo application (input/output/config assembly instances — the
§4.6 worked config's `100/150/151` layout; **pin the exact instance ids and sizes from the
OpENer sample app at S1-push time and update §4.6 if they differ**), and builds trivially on
Linux. **Flag: there is no widely-published official OpENer container image** — the repo ships
source + CMake, so `test-infra/opener/Dockerfile` (a ~15-line build stage over the sample app)
is part of this design, not an off-the-shelf pull.

**Fallback (and CI path): the protocol crate's own `testserver` class-1 target**
(PROTOCOL-DESIGN §12.5, D-ENIP-14) — a scriptable target used by the crate's unit tests, runnable
as a tiny binary for adapter-level tests where a container is unavailable (Windows dev loops,
constrained CI). The E2E gate (§11.4) uses OpENer; `testserver`-only validation of push is a
**blocking validation gap** for release purposes, reported as such — our-code-to-our-code proves
plumbing, not conformance.

cpppo has no backplane and OpENer routes directly ⇒ both sim configs omit `slot`; the
`slot`/backplane path stays a lab-PLC concern (§12.4).

## 14. Limitations & deviations

Surfaced up front, per the fidelity contract. Each also appears (present-tense, no roadmap
language) in the user-facing docs where relevant.

1. **Push writes are not device-acknowledged.** Implicit I/O has no per-write CIP confirmation;
   a push `sb/write` `ok:true` means the value rides the next cyclic O→T frame (§7.3,
   `applied: "next-frame"`). Poll writes remain CIP-acked. (User-docs phrasing: state both
   behaviors as plain present fact.)
2. **No structures/UDTs, no strings.** Supported values are CIP elementary scalars and 1-D
   arrays thereof (§5.1), in both modes. Logix STRING is a UDT ⇒ unsupported; config validation
   rejects them; browse marks them `supported: false` (D-EIP-16).
3. **No CIP Multiple Service Packet batching** — one request per signal per cycle in poll mode
   (D-EIP-15); `requestsPerCycle` makes the cost observable. Large signal counts want fewer,
   larger poll groups at longer intervals — or push mode, which has no per-signal request cost.
4. **`sb/browse` is device-dependent.** The tag-list service is a Logix-family capability;
   generic CIP devices (incl. plain I/O adapters like OpENer) answer `BROWSE_UNSUPPORTED`.
   Program-scoped enumeration covers program-prefixed symbol paths (PROTOCOL-DESIGN §7.3);
   deeper nested-structure iteration is not promised in v1. Push discovers nothing by itself —
   the assembly layout is configuration, there is no wire-discoverable field map.
5. **No CIP Security / TLS.** EtherNet/IP here is plaintext TCP 44818 + UDP 2222; deploy on an
   isolated OT network segment. (Stated as plain fact in user docs.)
6. **Lab-hardware-only validation paths, declared:** the `slot`/backplane routing path,
   connected class-3 messaging, **multicast T→O consume**, and push against a real
   PLC/remote-I/O device — cpppo has no chassis/class-1, and OpENer exercises P2P direct
   connections. Until real hardware passes §12.4's lab row, treat these config paths as
   implemented-but-hardware-unvalidated. Declared validation gaps, not silent ones. The
   extended-port (> 14) route encoding is additionally **disabled** pending a captured
   conformance vector (PROTOCOL-DESIGN D-ENIP-13).
7. **`sb/pause`/`sb/resume` extend the southbound contract** (D-EIP-3): not in `SOUTHBOUND.md`
   §2.2 and absent from both reference adapters. Candidate for core promotion; core is not
   edited by this work.
8. **Deviations vs the references, with rationale:** no subscription/monitored-item model
   (push/class-1 is the protocol's native cyclic model and is supported instead; OPC UA's
   filtered-subscription semantics do not transfer); no regex include/exclude signal matching
   (declaration-based like Modbus, D-EIP-7); allow-list `writes.allow[]` instead of Modbus's
   legacy `write.enabled` boolean (strictly stronger, D-EIP-5); `sb/subscribe-preview` from
   §2.2 is N/A (no dynamic subscription spec to preview — push connections are config-declared)
   and is deliberately not registered.
9. **Pause state does not survive restart** (in-memory; D-EIP-14/D-EIP-20). `sourceTs` is never
   emitted (D-EIP-11).
10. **One mode per instance** (D-EIP-2): a device needing both poll and push telemetry is two
    instances (they share nothing but the target device; each has its own session/connection).
